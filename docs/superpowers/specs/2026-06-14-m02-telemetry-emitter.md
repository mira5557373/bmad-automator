# M02 — Telemetry emitter and reader

## Context

M02 is the second milestone of the bmad-automator port and the operational counterpart to M01's typed event vocabulary. Where M01 landed the pure-data layer (the `Event` base, the 13 concrete event dataclasses, `UnknownEvent`, and `parse_event`), M02 lands the runtime that actually writes those events to disk and the read-side aggregations that consume them. Concretely, M02 introduces a `TelemetryEmitter` class that serializes `Event` instances to a JSONL file with cross-process locking via `filelock` plus an in-process `threading.Lock`, a `TelemetryReader` that streams the JSONL back through `parse_event` and exposes typed aggregations (per-epic cost rollups, per-story attempt counts, retro inputs), and the wiring of the existing log sites in `skills/bmad-story-automator/src/story_automator/commands/orchestrator.py`, `skills/bmad-story-automator/src/story_automator/commands/orchestrator_epic_agents.py`, and `skills/bmad-story-automator/src/story_automator/core/tmux_runtime.py` so that every place that today writes free-form log lines instead emits a typed event from the M01 catalog. The companion design doc records the locking strategy, the rotation policy, and the failure-mode taxonomy. M02 unblocks every later milestone that depends on telemetry being on disk in a typed form.

## Out of scope

M02 does not touch the typed-event vocabulary itself (that is M01 and is locked) and it does not add new event types — adding a new event would imply a new lifecycle hook and is explicitly deferred. The `cost_usd` capture path that turns `claude -p` stdout into a `CostCharged` event is M03. HMAC chaining of the JSONL stream for tamper-evidence is M04. The frontmatter-mutation rewrite in `skills/bmad-story-automator/src/story_automator/commands/state.py` is M05. The retry/escalation decision engine that consumes `RetryAttempt` and `EscalationTriggered` events is M06. The failure-classification taxonomy that fills the `error_class` field is M07. Budget alert threshold detection and percentage crossing logic are M08. Retrospective summary generation that consumes `RetroFired` is M09. Tmux runtime hardening (capture-pane buffer management, crash recovery) is M10. `CONTRIBUTING.md` updates and changelog tagging at `docs/changelog/YYMMDD.md` are M11. Plugin packaging is M12. The Node `bin/` shim work is M13. The `SECURITY.md` rewrite covering the threat model for the telemetry file is M14. M02 lands the emitter, the reader, and the wiring only — no semantics changes to existing events, no new events, no new helpers in `core/common.py`.

## Functional requirements

1. REQ-01 a new module must exist at `skills/bmad-story-automator/src/story_automator/core/telemetry_emitter.py` exporting a `TelemetryEmitter` class that must be importable under Python 3.11, 3.12, 3.13, and 3.14 without errors.
2. REQ-02 the `TelemetryEmitter` constructor must accept a `path: str | Path` argument and must lazily create the parent directory via `ensure_dir` from `story_automator.core.common` on the first `emit` call without raising if the directory already exists.
3. REQ-03 the `TelemetryEmitter.emit(event: Event)` method must append the result of `event.to_json_line()` followed by a single `\n` to the configured path, must hold a `filelock.FileLock` on `<path>.lock` for the duration of the append, and must additionally hold an instance-level `threading.Lock` to serialize concurrent emits from the same process.
4. REQ-04 the `TelemetryEmitter.emit` method must call `os.fsync` on the file descriptor before releasing the lock so a crash between `emit` calls cannot leave a partially written line on disk.
5. REQ-05 the `TelemetryEmitter` must accept an optional `run_id: str | None = None` constructor argument; when provided it must be stamped onto every emitted event whose `run_id` field is empty, and when omitted the emitter must pass the `Event` instance through unchanged so the caller's explicit `run_id` always wins.
6. REQ-06 a new module must exist at `skills/bmad-story-automator/src/story_automator/core/telemetry_reader.py` exporting a `TelemetryReader` class whose constructor accepts a `path: str | Path` argument and whose `iter_events()` method must yield `Event` instances by reading the file line-by-line and dispatching each non-empty line through `parse_event` from `story_automator.core.telemetry_events`.
7. REQ-07 the `TelemetryReader.iter_events` method must skip blank lines silently and must propagate `json.JSONDecodeError` from `parse_event` so corruption is loud rather than silent; the method must return an empty iterator when the file does not exist rather than raising `FileNotFoundError`.
8. REQ-08 the `TelemetryReader` must expose a `cost_by_epic() -> dict[str, float]` aggregation that sums `cost_usd` across every `CostCharged` event grouped by the `epic` field, a `attempts_by_story() -> dict[tuple[str, str], int]` aggregation that counts `RetryAttempt` events keyed by `(epic, story_key)`, and a `retro_inputs(epic: str) -> dict[str, Any]` aggregation that returns the `stories_completed`, `total_cost_usd`, and `duration_s` fields from the most recent `RetroFired` event matching the supplied epic or an empty dict if none exists.
9. REQ-09 every existing log site in `skills/bmad-story-automator/src/story_automator/commands/orchestrator.py` that today emits a story lifecycle log line must be replaced by a single `TelemetryEmitter.emit` call constructing the matching M01 event type with `iso_now()` as the timestamp; the public function signatures of `orchestrator.py` must not change.
10. REQ-10 every existing log site in `skills/bmad-story-automator/src/story_automator/commands/orchestrator_epic_agents.py` that today emits a retry, escalation, or review cycle log line must be replaced by a `TelemetryEmitter.emit` call constructing `RetryAttempt`, `EscalationTriggered`, or `ReviewCycle` as appropriate; the existing free-form log lines must be removed in the same commit.
11. REQ-11 every existing tmux spawn and tmux completion log site in `skills/bmad-story-automator/src/story_automator/core/tmux_runtime.py` must be replaced by a `TelemetryEmitter.emit` call constructing `TmuxSessionSpawned`, `TmuxSessionCompleted`, or `TmuxSessionCrashed` as appropriate based on the existing branch that distinguishes normal exit from abnormal exit.
12. REQ-12 a new test file at `tests/test_telemetry_emitter.py` must contain unit tests covering single-event emit, multi-event emit, blank-line handling on read, the fsync-before-unlock ordering invariant, the run-id stamping behavior, and the round-trip equivalence of every M01 event type through emit then read; the tests must use `unittest.TestCase` and must run cross-platform without invoking tmux or subprocesses.
13. REQ-13 a new test file at `tests/test_telemetry_reader.py` must contain unit tests covering the three aggregation methods, the empty-file case, the missing-file case, the malformed-line propagation case, and the mixed-event-type aggregation case where only the relevant event types contribute to each rollup; the tests must use temporary directories via `tempfile.TemporaryDirectory`.
14. REQ-14 the two new modules and the wiring edits must use only the Python standard library plus `filelock` and `psutil`; when grepped for new imports the new files must yield zero matches outside that allowlist, and the wiring edits to existing files must not introduce any third-party import not already present.
15. REQ-15 the new modules must import `iso_now`, `compact_json`, and `ensure_dir` from `story_automator.core.common` where needed and must not duplicate any helper that already exists in that module.

## Non-functional requirements

- Each new module's file size is no more than 500 source lines of code excluding tests and docstrings, per the project's `CONTRIBUTING.md` guideline.
- All public classes, methods, and constructor arguments carry PEP 604 union-typed annotations; both new modules begin with `from __future__ import annotations`.
- The new modules and their test files pass `ruff check` and `ruff format --check` under the project's existing ruff configuration with no additional rule disables.
- Line coverage of `core/telemetry_emitter.py` and `core/telemetry_reader.py` measured by `coverage` meets or exceeds 85 percent each.
- The test suite runs on Windows git-bash, WSL Ubuntu, and Linux CI without modification; tests use `tempfile.TemporaryDirectory` for file paths and must not invoke tmux, must not call any subprocess, and must complete with total wall-clock under three seconds.
- The emit path is safe under cross-process concurrency: two processes writing to the same telemetry path must not interleave partial lines, validated by an in-process multi-thread stress test that exercises the same lock acquisition order.
- The reader is streaming and must not load the full file into memory; `iter_events` must yield one parsed `Event` per line without buffering more than a single line at a time.
