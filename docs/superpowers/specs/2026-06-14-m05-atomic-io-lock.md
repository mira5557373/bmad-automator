## Context

Milestone M05 (atomic-io-lock) replaces the existing regex-driven frontmatter mutation logic in `skills/bmad-story-automator/src/story_automator/commands/state.py` with a hardened atomic input/output and run-lock substrate. Today, `cmd_build_state_doc` performs in-place text rewrites against `output_path.write_text(text)` after sequential `re.sub` passes; concurrent orchestrator invocations can interleave writes, partially clobber YAML frontmatter, or leave a half-written file on crash. The legacy marker-file approach (a sentinel under the run directory) also cannot detect a stale PID held by a previous host or distinguish a crashed run from an in-flight one. M05 introduces a new module `skills/bmad-story-automator/src/story_automator/core/atomic_io.py` providing a per-path `threading.Lock` registry, a tmp-file + `os.replace` write helper with Windows retry semantics, and a `filelock.FileLock`-backed run lock whose payload is a composite identity record (`pid`, `start_time`, `hostname`, `heartbeat_iso`, `run_id`). A daemon `HeartbeatThread` refreshes the heartbeat field every 60 seconds; consumers may treat any lock whose heartbeat is older than 600 seconds (and whose PID is no longer alive per `psutil.pid_exists`) as stale and reclaim it. M05 is independent of other in-flight milestones and may land before or after them, but it is classified HIGH risk because it rewrites a hot path inside `state.py` that the orchestrator and retro commands call on every run.

## Out of scope

M05 does not touch the typed telemetry event substrate delivered by M01 in `skills/bmad-story-automator/src/story_automator/core/telemetry_events.py`, nor the emitter and wiring work owned by M02 against `skills/bmad-story-automator/src/story_automator/commands/orchestrator.py`, `skills/bmad-story-automator/src/story_automator/commands/orchestrator_epic_agents.py`, and `skills/bmad-story-automator/src/story_automator/core/tmux_runtime.py`. The cost-capture path (M03), HMAC chaining of telemetry batches (M04), failure-classification consumers (M07), retro persistence reshaping (M08), agent-config snapshot refactor (M09), tmux session reconciliation (M10), and the operator-doc edits to `CONTRIBUTING.md` (M11) and `SECURITY.md` (M14), along with the retroactive changelog tagging under `docs/changelog/YYMMDD.md` (also M11), all remain outside M05. M05 also explicitly excludes any new third-party dependency beyond the existing allowlist; in particular no cross-process queue, no SQLite-backed lock, and no platform-specific advisory-lock module. Heartbeat tuning knobs (interval, stale window) are constants for this milestone — exposing them as policy fields is a follow-up for M12. Migration of any existing marker-file artifacts found on operator disks is one-shot best-effort and does not block startup; reverse-compatibility shims for the regex frontmatter rewriter are not provided beyond preserving observable output equality on the happy path.

## Functional requirements

1. REQ-01 The module `skills/bmad-story-automator/src/story_automator/core/atomic_io.py` must exist and must declare `from __future__ import annotations` as its first non-docstring statement.
2. REQ-02 `atomic_io.py` must expose a public function `write_atomic_text(path: Path, data: str, *, encoding: str = "utf-8") -> None` that writes to a sibling temp file (suffix `.tmp-<pid>-<monotonic_ns>`) in the same directory as `path`, fsyncs the temp file, then performs `os.replace(tmp, path)` so that readers observe either the previous contents or the full new contents.
3. REQ-03 `write_atomic_text` must acquire a per-path `threading.Lock` from an internal registry keyed by the resolved absolute path before performing the write, and must release the lock in a `finally` block; the registry itself must be guarded by a module-level `threading.Lock` so lock acquisition is thread-safe.
4. REQ-04 On Windows, when `os.replace` raises `PermissionError` (errno-equivalent of ERROR_SHARING_VIOLATION), `write_atomic_text` must retry up to 5 times with an exponential backoff of 50ms, 100ms, 200ms, 400ms, 800ms, and must raise the final `PermissionError` if all retries fail; on POSIX, `os.replace` must be called exactly once.
5. REQ-05 `atomic_io.py` must expose a `@dataclass(kw_only=True)` named `RunLockIdentity` with fields `pid: int`, `start_time: float`, `hostname: str`, `heartbeat_iso: str`, `run_id: str`, and a method `to_json(self) -> str` that returns a deterministic compact JSON encoding via `compact_json` imported from `skills/bmad-story-automator/src/story_automator/core/common.py`.
6. REQ-06 `atomic_io.py` must expose `acquire_run_lock(lock_path: Path, *, run_id: str, timeout: float = 0.0) -> RunLockHandle` that wraps `filelock.FileLock(str(lock_path) + ".lock")`, writes a `RunLockIdentity` JSON payload to `lock_path` after acquisition using `write_atomic_text`, and returns a context-manager handle releasing both the FileLock and the payload file on `__exit__`.
7. REQ-07 `acquire_run_lock` must populate `heartbeat_iso` and `start_time` using `iso_now` imported from `core/common.py` and `time.time()` respectively, and must populate `hostname` from `socket.gethostname()` and `pid` from `os.getpid()`.
8. REQ-08 The module must expose a `HeartbeatThread` class subclassing `threading.Thread` with `daemon=True`, an `interval` constant of 60.0 seconds, a `stop()` method that sets a `threading.Event`, and a `run()` loop that rewrites the lock payload via `write_atomic_text` with a refreshed `heartbeat_iso` until the stop event is set.
9. REQ-09 The module must expose `is_stale(identity: RunLockIdentity, *, now: float | None = None) -> bool` returning `True` only when `(now or time.time()) - parse_iso_seconds(identity.heartbeat_iso) > 600.0` and `psutil.pid_exists(identity.pid)` returns `False`.
10. REQ-10 `commands/state.py` must be updated so every call site that previously invoked `output_path.write_text(text)` instead routes through `write_atomic_text`, preserving the existing JSON envelope returned by `write_json` and the existing return codes.
11. REQ-11 The previous marker-file sentinel pattern used by `state.py` and its callers must be removed; the new `acquire_run_lock` API must be used in its place, and any legacy marker discovered at startup must be deleted with a single best-effort `Path.unlink(missing_ok=True)` call.
12. REQ-12 All public functions and classes in `atomic_io.py` must carry type annotations, must not import any third-party module other than `filelock` and `psutil`, and the module must remain under 500 lines of source code per the project file-size guardrail.
13. REQ-13 Unit tests under `tests/test_atomic_io.py` must cover: atomic replace happy path, simulated `PermissionError` retry on Windows (via `unittest.mock.patch`), per-path lock serialization across two threads, `RunLockIdentity` round-trip JSON, `HeartbeatThread` refresh observed by reading the payload twice, and `is_stale` returning `False` for a live PID and `True` for a synthetic dead PID with an aged heartbeat.
14. REQ-14 Tests must use `unittest.TestCase`, must not spawn subprocesses, must not depend on tmux, and must pass on Windows git-bash, WSL Ubuntu, and Linux CI without modification.

## Non-functional requirements

- Reliability: a process crash between the temp-file fsync and the `os.replace` step must leave the original target file intact and readable; readers in a parallel process must never observe a truncated frontmatter document.
- Concurrency safety: two threads in the same process writing to the same resolved path must serialize through the per-path registry, and two processes writing to the same path must serialize through `filelock.FileLock` with no deadlock when timeouts expire.
- Portability: the module must function identically on CPython 3.11, 3.12, 3.13, and 3.14 across Windows (NTFS), macOS (APFS), and Linux (ext4/btrfs); no `os.O_TMPFILE`, `fcntl`, or `msvcrt` calls — only cross-platform stdlib primitives and the two allowed third-party packages.
- Observability: lock acquisition failures and Windows retry exhaustion must raise typed exceptions (`RunLockBusy`, `AtomicWriteRetryExhausted`) so that future M02 telemetry wiring can classify them without string-matching.
