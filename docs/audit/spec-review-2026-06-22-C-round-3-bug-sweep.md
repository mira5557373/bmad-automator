# Spec Review — Milestone C (Round-3 bug sweep, lenses K/L/M)

**Reviewer:** adversarial gap-and-enhancement pass
**Reviewed artifacts:**
- `docs/superpowers/specs/2026-06-22-round-3-bug-sweep-design.md`
- `docs/superpowers/plans/2026-06-22-round-3-bug-sweep-plan.md`
**Branch:** `bma-d/integration-all` · **HEAD at review:** `5424b7e` (NOT `6a957d2` as the spec claims)
**Baseline test count (measured):** 4070 passing, 2 skipped — only with `PYTHONPATH=skills/bmad-story-automator/src`. Without it: 28 import errors, 3665 collected.
**Verdict:** **needs-enhancement** (does not need a full redesign, but cannot execute as written).

## TL;DR

The spec articulates the *philosophy* of a triage-capped bug sweep well — fresh lenses K/L/M, ≤5 fix-now cap, "information quality not patch volume" framing, and explicit anti-goals are all genuinely useful guardrails that round-2 lacked. The plan, however, is **not executable as written** for three independent mechanical reasons: (a) every `python3 -m unittest discover -s tests` invocation in the plan is missing `PYTHONPATH=skills/bmad-story-automator/src`, which means the very first pre-requisite step will collapse with 28 `ModuleNotFoundError: No module named 'story_automator'` errors and look like a 405-test regression; (b) the spec and plan reference `tests/test_frozen_gate_surface.py` as the contract-verifier for the frozen-gate-surface, but that file does **not exist** in the repo — frozen surface is currently only a markdown doc (`docs/spec/frozen-gate-surface.md`); (c) the HEAD pin (`6a957d2`) is one commit behind the actual branch HEAD (`5424b7e`), and the "find me at or after `6a957d2`" instruction was written before milestones A/B/D specs landed. Beyond those three mechanical blockers, there are deeper triage-rubric gaps: HIGH vs MED vs LOW are defined narratively but never operationalized into a per-lens ordinal rubric a reviewer could replay; the adversarial-verification step is essentially "the implementer is honest" (rounds 1+2 downgraded ~30% of findings, and round-3 has *no* second reader in the loop); the 5-fix cap is enforced only by self-discipline (Phase 2's `C2.4` "demote excess" is the implementer's own checkbox); and "no high-confidence findings" → `[SKELETON]` outcome is a perverse incentive that the spec does not defend against. There is also a non-trivial Milestone-B interaction: B1+B2 modify `core/evidence_io.py` and `core/gate_orchestrator.py` — the same two modules at the centre of Lens K and Lens M — and the spec is silent on the merge order, baseline drift, and "did B1 just fix one of C's findings?" question. Sixteen findings below (3 HIGH, 8 MED, 5 LOW); after the recommended enhancements, this milestone is safe to execute.

## HIGH-severity findings (one section per HIGH finding)

### C-H-01 · Plan PYTHONPATH omission breaks every test step

**File:section:** `docs/superpowers/plans/2026-06-22-round-3-bug-sweep-plan.md` §Pre-requisites (lines 11-15), §C3.N.2 (line 177), §C3.N.4 (line 181), §C3.N.5 (line 183), §C3.N.8 (line 195), §C4.2 (lines 227-230).

**Problem:** The project's `package.json` defines `"test:python": "PYTHONPATH=skills/bmad-story-automator/src python3 -m unittest discover -s tests"` precisely because the source layout (`skills/bmad-story-automator/src/story_automator/…`) is not on `sys.path` by default. I reproduced this: a bare `python3 -m unittest discover -s tests` produces **28 `ModuleNotFoundError: No module named 'story_automator'` errors** and a "3665 ran, 28 errors" report; with `PYTHONPATH=skills/bmad-story-automator/src` it cleanly reports "4070 ran, 0 failures, 2 skipped". The plan's Pre-requisite step #2 (line 12) is literally the bare form; the implementer following the plan will see 405 missing tests at step one and think "the baseline isn't green — I need to fix that first" — burning their first hour on a non-existent regression. Every subsequent verification step in the plan has the same bug.

**Suggested patch (concrete):** In every `python3 -m unittest …` line in the plan, prepend `PYTHONPATH=skills/bmad-story-automator/src`, OR replace the entire test invocation with `npm run test:python` (which already injects PYTHONPATH). For the focused `-k test_bugfix_c_<n>` filter at §C3.N.2 and §C3.N.4: `PYTHONPATH=skills/bmad-story-automator/src python3 -m unittest discover -s tests -k test_bugfix_c_<n>`. Add a one-line note at the top of §Pre-requisites: "All `unittest` commands assume `PYTHONPATH=skills/bmad-story-automator/src` (matches `npm run test:python`)."

### C-H-02 · `tests/test_frozen_gate_surface.py` does not exist

**File:section:** Spec §5.2 line 162 ("Verified by `tests/test_frozen_gate_surface.py` staying green"); §6.3 line 190 ("`tests/test_frozen_gate_surface.py` → green"); Plan §Pre-requisites line 15, §C3.N.8 line 195, §C4.2 line 230, §Rollback line 297, §Definition-of-done line 328.

**Problem:** I checked. `tests/test_frozen_gate_surface.py` is not present in the repository. The only frozen-gate-surface artifact is the markdown doc `docs/spec/frozen-gate-surface.md`, which lists symbols but is not executable. The acceptance criterion "Frozen-gate-surface: zero diff to the public symbol roster" therefore has no automated check — the implementer would either (a) skip it (silently accept any symbol drift), (b) write the test mid-milestone (a non-trivial scope expansion that's not budgeted), or (c) interpret it as "no diff to `docs/spec/frozen-gate-surface.md`" (the markdown doc, which is a far weaker promise than a symbol-roster import test). The plan's `Rollback` section explicitly says "If `tests/test_frozen_gate_surface.py` goes red, a public symbol was accidentally moved" — but a test that doesn't exist can never go red. This is the same shape of bug as round-2's D-04 follow-up: a verifier referenced in the spec that was never written.

**Suggested patch (concrete):** Pick one of two paths in the spec rewrite:
- **Path A (recommended):** Author `tests/test_frozen_gate_surface.py` as a precondition (before any sweep starts). The test reads `docs/spec/frozen-gate-surface.md`, parses the listed public symbols (module-by-module), imports each module, and asserts every listed symbol exists and is callable / a class / a Protocol / has the expected attribute. ~80 LOC test file, one-off effort. Add a new Phase 0 task to the plan: "C0.1 — author `tests/test_frozen_gate_surface.py` and confirm green against current HEAD". Note this is **separate** scope from C's three lenses — it's prerequisite plumbing.
- **Path B (cheaper):** Drop every reference to `tests/test_frozen_gate_surface.py` from the spec and plan; replace the acceptance criterion with `git diff <milestone-start-sha>..HEAD -- docs/spec/frozen-gate-surface.md \| wc -l == 0` AND a manual import-roster smoke (a 6-line ad hoc Python script that imports the listed symbols, recorded in §0 of the audit report). Weaker but honestly verifiable.

### C-H-03 · HEAD pin (`6a957d2`) is stale

**File:section:** Spec §1 line 13 ("HEAD `6a957d2`"); Plan §Pre-requisites line 9 ("confirm `git log -1 --oneline` reads `6a957d2`").

Add a serialisation constraint to spec §6: "C executes either *strictly after* B's `milestone-b-operability-batch` tag exists, or *strictly before* B's first commit lands." Spec §3.1's LOC table becomes "snapshot at milestone-C-start (recorded in §0)" rather than fixed numbers. Add a §C1.0 task: "snapshot HEAD SHA + per-module LOC into §0 of the audit report; this is the reference for all later LOC comparisons." Update §11's "Quick LOC + scope estimate" to clarify that the per-module pre-sweep LOC may have drifted from §3.1's prose due to B.

## Verdict

**needs-enhancement** → **enhancements applied (2026-06-22)**

The spec's philosophy is right and the plan's structure is right; the milestone is *not* a redesign candidate. But three independent HIGH-severity mechanical issues (PYTHONPATH omission, non-existent frozen-surface test, stale HEAD pin) mean the implementer following the plan will produce a corrupted audit-report §0 (wrong LOC, wrong SHA, wrong test count) within the first hour. Apply enhancements 1-7 above, then this is safe to execute. The MED-severity findings (rubric, adversarial verifier, B↔C ordering, lens-M coverage) raise the *quality* of the sweep substantially — they are not strictly blocking, but skipping them means round-3's lessons-learned will mirror round-2's ("self-triage drift", "cap was soft, we shipped too many"). The LOW-severity findings are pure backlog.

Estimated total enhancement cost: 30-60 minutes of spec editing. Estimated downstream cost of skipping the enhancements: 1-2 wasted hours per implementer + the same round-2 over-shipping pattern repeating.
