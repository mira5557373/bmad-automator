### Detailed bug-by-bug breakdown

#### Bug 1 — L3: worst-of metric aggregation (`a86c216`)

**File:** `core/checks/category_rules.py` (`_aggregate_metrics`)
**Severity:** HIGH — fail-open in multi-collector category rules
**Confidence:** 5/5 verified by adversarial reproducer

Previously `_aggregate_metrics` returned the first record carrying the requested key. For multi-collector categories (`security`, `correctness`, `test_quality`, `license`), a clean record from tool A could silently mask a violating record from tool B. Reducer table is now keyed on metric name:
- **sum:** `sast_high_count`, `deps_critical_count`, `secrets_count`, `regressions`, `flaky_count`, `hard_wait_count`, `forbidden_count`, `boundary_violations`, `mutants_*`
- **min:** `coverage_pct`, `mutation_score`, `test_review_score`, `scenarios_passed`
- **max:** `rto_seconds`, `rpo_seconds`, `pod_cost_per_tenant`, `scenarios_total`
- **any:** `slo_breached`
- **all:** `signal_survived`, `rollout_completed`
- **default (unknown keys):** `max` — most conservative for counts

Regression coverage: `tests/test_bugfix_L3.py` covers direct reducer behavior and the four multi-collector category rules.

#### Bug 2 — L4: empty / all-NA verdict aggregation (`2c9bc84`)

**File:** `core/gate_rules.py` (`aggregate_verdicts`)
**Severity:** HIGH — fail-open in the verdict engine
**Confidence:** 5/5 verified by adversarial reproducer

`aggregate_verdicts({})` and `aggregate_verdicts({"a": "NA"})` previously returned PASS, violating the §6.3 fail-closed invariant. A gate where every category was NA (or none had been evaluated at all) was being approved as if the factory had run. After filtering NA, an empty active set now returns FAIL with rationale. Two existing tests (`test_all_na_is_pass`, `test_all_na_categories_pass`) encoded the buggy behavior and were updated to assert FAIL. New regression at `tests/test_bugfix_L4.py` covers empty input, all-NA input, and the mixed case (one PASS plus NAs still PASSes).

#### Bug 3 — L5: naive ISO timestamps in waivers (`60f3baf`)

**Files:** `core/gate_rules.py` (`_parse_iso`), `core/gate_schema.py` (`validate_waiver`, new `_require_tz_aware_iso` helper)
**Severity:** HIGH — orchestrator crash on otherwise-valid input
**Confidence:** 5/5 verified by adversarial reproducer

`_parse_iso` previously returned a naive datetime when the ISO string carried no timezone (e.g. `"2026-06-20T00:00:00"`). The subsequent `now >= expires` comparison inside `is_waiver_expired` raised a bare `TypeError("can't compare offset-naive and offset-aware datetimes")` that propagated out of `check_gate_reuse` and crashed the orchestrator, instead of producing the documented `WaiverError`. Now `_parse_iso` raises `ValueError` for naive timestamps; existing callers that already catch `ValueError` (`is_waiver_expired` → `WaiverError`, `_check_ttl` → `(False, reason)`) surface a clean error. `validate_waiver` additionally rejects naive `issued_at`/`expires_at` at schema-load time. Regression in `tests/test_bugfix_L5.py`.

#### Bug 4 — LENS-B-01: nonce-guard lockfile cleanup (`dcd82d5`)

**File:** `core/deferred_work.py`
**Severity:** HIGH — silent mutual-exclusion violation
**Confidence:** 5/5 verified by adversarial reproducer

Previously a steal-victim's `__exit__` unconditionally unlinked the lockfile — which by then belonged to the peer that had stolen it after the steal-threshold mtime check passed. This defeated mutual exclusion around the read-modify-write inside `append_entry` and risked lost entries in `deferred-work.md`. Now a per-instance `uuid4+pid` token is written into the lockfile on `__enter__`, and `__exit__` only unlinks when the on-disk token still matches its own. Stdlib-only (no new deps). 6 regression tests at `tests/test_bugfix_LENS_B_01.py` cover token presence, distinct nonces per acquisition, stolen-lock survival, normal cleanup, roundtrip ordering, and `append_entry` integration after a stale-lock steal.

#### Bug 5 — E2_C9_D-SEC-04: dispatcher env contamination (`880a7d0`)

**File:** `core/cli_dispatcher_invokers.py` (`claude_code_invoker`)
**Severity:** HIGH (single-user threat model: MED) — cross-session env leakage
**Confidence:** 5/5 verified by adversarial reproducer

`claude_code_invoker` mutated `os.environ` with `BMAD_AUTO_STORY_KEY`, `BMAD_AUTO_PHASE`, `BMAD_AUTO_CLI_ID`, `BMAD_AUTO_COMMIT_SHA`, and `BMAD_AUTO_TASK_ID` and never restored prior values, contaminating the parent process between sequential dispatcher invocations. Cleanup code and attribution probes reading `os.environ` between sessions saw stale keys from the previous session. Now: snapshot prior values with an absent-sentinel, apply the mutation only across the `_spawn_session_hook` call (so the subprocess still inherits keys for `tmux new-session -e KEY=VAL`), then restore in a finally — including removing keys that were absent beforehand and even on spawn exception. 4 regression tests at `tests/test_bugfix_E2_C9_D_SEC_04.py`.

## Adversarial-verifier refutations (what was reported but was not actually a bug)

### REFUTED-1 — "`_aggregate_metrics` always returns wrong value"

**Original claim:** Bug L3 was framed as "the function always returns the first record's value regardless of key".

**Verifier finding:** False. The first-wins lookup was keyed on the *record carrying the requested key*, not the first record. So if the requested key only existed on the second collector's record, the second record was returned. The actual defect is narrower: when *multiple* collectors emitted the same key, the first one won and the rest were dropped. Fix scope was tightened accordingly (per-metric reducer table) rather than rewriting the function.

### REFUTED-2 — "`_claude_completion_marker_present` matches any line ending in 'ed'"

**Original claim:** Marker regex `\\w+ed` would match "tested" anywhere in the buffer and prematurely finalize the session.

**Verifier finding:** Partially refuted. The regex is anchored by `(?:^|✻\\s+)` with `re.M`. The "Tested for 3m 12s" false-positive reproducer in `2026-06-19/index.md` F-036 *does* trigger because `^` with `re.M` matches the start of any line — but the production code path only feeds the *most recent tmux pane snapshot* through this matcher, and the spinner-glyph false positive only fires when the snapshot's first line begins with a past-tense verb. Real-world frequency is low (no production hit recorded). Downgraded to LOW severity and left as tracked debt.
