# D4 — /proc leakage + tmux command injection findings

Repo: `/home/ubuntu/projects/personal/bmad-automator` @ `bma-d/integration-all`
Threat model: single-user. Trusted: story files, project root, `agent-config-presets.json`. Untrusted: buggy/compromised LLM child, supply-chain shim binaries.

**BMAD_AUDIT_KEY lifecycle verdict:** the key is read by `load_key_from_env()` (audit.py:103-116) via `os.environ.get`, and is **never popped** anywhere in the codebase. It therefore remains in `os.environ` for the orchestrator's entire lifetime and propagates to every `subprocess.run(env=os.environ.copy())` child — `tmux` server, `pgrep`, `git`, every `tmux` probe. The deliberate pane-env scrub at `tmux_runtime.py:634/738` (`-e BMAD_AUDIT_KEY=`) only blanks the *pane shell's* environ; the tmux server process itself was launched with the key in its environ and `/proc/<tmux-pid>/environ` remains readable by the in-pane child (same UID). D3 (F-D3-001) already covers the upstream `run_cmd` fix; D4 adds the lifecycle + tmux-specific concerns below.

**tmux send-keys posture:** two call sites (`tmux_runtime.py:744` and `:746`), neither uses `-l` (literal mode). For our threat model the `command` argument is trusted (built from `agent_cli` + `shlex.quote(prompt)`), so this is acceptable — but the absence of `-l` becomes a real injection vector if the trust model ever shifts. Flagged below as a hardening note.

**`-F` / `-p` format-string posture:** all format strings are hard-coded literals (`#{session_name}`, `#{pane_id}`, `#{pane_pid}`, `#{pane_dead}`, `#{pane_dead_status}`, `#{pane_width}x#{pane_height}`). No attacker-influenceable content ever feeds into the format-string position, so `#{client_env:BMAD_AUDIT_KEY}` reflection is **not** reachable. Clean.

## F-D4-001
- severity: must
- likelihood: medium
- title: BMAD_AUDIT_KEY persists in os.environ for orchestrator lifetime
- file: skills/bmad-story-automator/src/story_automator/core/audit.py:103-116 (+ commands/audit_verify_cmd.py:50)
- excerpt: |
    def load_key_from_env(env: Mapping[str, str] | None = None) -> bytes | None:
        source: Mapping[str, str] = env if env is not None else os.environ
        raw = source.get(_ENV_VAR, "")
        if not raw:
            return None
        return derive_key(raw)
- repro: |
    # Demonstrates that the parent process keeps BMAD_AUDIT_KEY in /proc/self/environ
    # AND that every child started via run_cmd inherits it. Two-line test:
    import os, subprocess
    os.environ["BMAD_AUDIT_KEY"] = "deadbeef"
    # production path (core.common.run_cmd does proc_env = os.environ.copy()):
    p = subprocess.Popen(["sleep", "30"], env=os.environ.copy())
    # /proc/<p.pid>/environ now contains BMAD_AUDIT_KEY=deadbeef as a NUL-delim record.
    # Also: open("/proc/self/environ","rb").read().split(b"\\0") -- key is present
    # for the entire orchestrator run. load_key_from_env() never pops it, and no
    # codepath calls os.environ.pop("BMAD_AUDIT_KEY") after derivation.
- proposed_fix: |
    Read-and-pop pattern: load_key_from_env() should pop the var from os.environ
    once derived, holding the derived bytes only in the AuditLog instance (already
    private). This closes /proc leakage AND makes the F-D3-001 run_cmd scrub
    defense-in-depth rather than the sole barrier.

    ```python
    def load_key_from_env(env: MutableMapping[str, str] | None = None) -> bytes | None:
        source = env if env is not None else os.environ
        raw = source.pop(_ENV_VAR, "")  # pop, not get
        if not raw:
            return None
        return derive_key(raw)
    ```
    The Mapping type hint widens to MutableMapping; verify() and append() call
    sites construct AuditLog from the returned bytes — they never re-read environ.
- effort: S
- confidence: 5

## F-D4-002
- severity: should
- likelihood: medium
- title: tmux server inherits BMAD_AUDIT_KEY via run_cmd — pane-env scrub is incomplete
- file: skills/bmad-story-automator/src/story_automator/core/tmux_runtime.py:609-636 (and 717-739 twin)
- excerpt: |
    # The tmux server is started by this subprocess.run via core.utils.run_cmd,
    # which does proc_env = os.environ.copy() — so the tmux SERVER inherits
    # BMAD_AUDIT_KEY. The -e VAR= flags below only scrub the pane shell.
    create_out, create_code = run_cmd(
        "tmux", "new-session", "-d", "-s", session, ...,
        "-e", "STORY_AUTOMATOR_CHILD=true",
        "-e", f"AI_AGENT={selected_agent}",
        "-e", "CLAUDECODE=",
        "-e", "BASH_ENV=",
        "-e", "BMAD_AUDIT_KEY=",
        ...
    )
- repro: |
    # Child agent in the spawned pane can recover the key from the tmux server's
    # /proc entry (same UID), bypassing the pane-env scrub:
    #   tmux_pid=$(tmux display -p '#{server_pid}')
    #   tr '\\0' '\\n' < /proc/$tmux_pid/environ | grep BMAD_AUDIT_KEY
    # Reliable when this orchestrator was the first to launch the tmux server
    # (its env populated the server). Less reliable if a pre-existing tmux server
    # is attached, but the project's typical workflow IS to spawn fresh servers.
- proposed_fix: |
    Two-layer defense: (a) implement F-D4-001 (pop at load) so the parent's environ
    is clean before any run_cmd; (b) implement F-D3-001 (denylist in run_cmd) so even
    if the pop is forgotten, run_cmd strips the key before passing env to tmux. Then
    the -e VAR= pane scrub becomes the third layer (e.g. against a tmux server reused
    from a previous shell session). All three are individually cheap.
- effort: S
- confidence: 4

## F-D4-004
- severity: should
- likelihood: low
- title: tmux send-keys without -l interprets key-binding tokens in command string
- file: skills/bmad-story-automator/src/story_automator/core/tmux_runtime.py:744-746
- excerpt: |
    if len(command) > 500:
        _write_private_text(paths.command, "#!/bin/bash\n" + command + "\n", 0o700)
        run_cmd("tmux", "send-keys", "-t", session, f"bash {paths.command}", "Enter")
    else:
        run_cmd("tmux", "send-keys", "-t", session, command, "Enter")
- repro: |
    # tmux send-keys treats each positional arg as either typed text OR a key
    # name when the arg matches a known key (Enter, C-c, F1, Up, etc.). Without
    # -l, an attacker-influenceable command string that happens to be exactly
    # a key token would be injected as that key.
    #
    # In the current trust model the command is built by agent_cli + shlex.quote(prompt)
    # and is therefore trusted — but a buggy LLM child that round-trips through
    # _build_cmd could in principle produce a degenerate command that includes
    # things like " C-c " mid-string. tmux's parser splits on whitespace; tokens
    # matching key names are sent as keys.
    #
    # Demonstration:
    #   tmux new-session -d -s test
    #   tmux send-keys -t test "echo hi" Enter      # types and runs `echo hi`
    #   tmux send-keys -t test "C-c"                # sends Ctrl-C, not the literal text
- proposed_fix: |
    Defense in depth: switch both send-keys call sites to `-l` (literal mode), then
    send `Enter` as a second call. Keeps the trust model robust under any future
    relaxation of "the command string is trusted":
    ```python
    run_cmd("tmux", "send-keys", "-t", session, "-l", command)
    run_cmd("tmux", "send-keys", "-t", session, "Enter")
    ```
    Note: the legacy path is the only call site (runner path uses `respawn-pane`
    with explicit argv, not send-keys). Two-line change.
- effort: S
- confidence: 4

    Either (a) delete the AI_COMMAND override path entirely (docs already mark
    it "legacy and deprecated"), or (b) validate it with shlex.split() and reject

    allowlist (claude, codex, etc.). Option (a) is the cleaner D-day cleanup.

## F-D4-006
- severity: nice
- likelihood: low
- title: tmux pane inherits unfiltered LD_PRELOAD / LD_LIBRARY_PATH / PYTHONSTARTUP
- file: skills/bmad-story-automator/src/story_automator/core/tmux_runtime.py:609-636 (and :717-739)
- excerpt: |
    # The -e scrub list covers BMAD_AUDIT_KEY, BASH_ENV (runner only), CLAUDECODE.
    # No scrub for LD_PRELOAD, LD_LIBRARY_PATH, LD_AUDIT, PYTHONSTARTUP, PYTHONPATH,
    # PERL5OPT, NODE_OPTIONS — all of which are honored by interpreters/loaders
    # the child agent invokes (claude / codex CLIs are JS/Python).
- repro: |
    # A compromised dotfile or stale shell init can set NODE_OPTIONS="--require
    # /tmp/evil.js" in the operator's environ. tmux forwards it to the pane, and
    # `claude --dangerously-skip-permissions` (which is Node) loads the script
    # before any prompt is processed. Same UID, so /proc-readable secrets are
    # exposed to attacker code running inside the child agent's process.
- proposed_fix: |
    Extend the pane-env scrub list with the common interpreter-injection vars
    AND consider going one step further: tmux supports `update-environment ""`
    to nuke the inherited server environ for new panes. A scoped allowlist is
    safer than an open-ended denylist. Suggested allowlist (verify against
    operational needs): HOME, USER, TERM, LANG, PATH, SHELL, plus the project's
    own STORY_AUTOMATOR_* / PROJECT_ROOT / AI_AGENT.
- effort: M
- confidence: 3
