# D10 — Operability for a solo operator

Audit target: bmad-automator @ branch `bma-d/integration-all`
Operator persona: `mira` running on a personal VPS, no on-call buddy.

## Summary

- **Doctor command:** ABSENT. `python3 -m story_automator doctor` exits 1
  ("Unknown command: doctor"). No env/tool preflight at all on the runtime
  side; the only `verify-setup.sh` in repo root targets an unrelated `sw`
  tool, not bmad-automator.
- **README quickstart:** Walks cleanly. `npx bmad-story-automator`,
  `story-automator --version|--help`, and `npm run lint:python | test:python |
  verify` all map to real scripts in `package.json`. No finding.
- **Top-3 failure-mode runbooks:** 1 of 3 fully covered, 1 partial, 1 missing.
  - "story stuck in pane" — covered (`operations.md` "A stuck or wedged child
    session" + `troubleshooting.md` "Stale tmux Sessions").
  - "audit chain broken on verify" — **detection only, no key-rotation
    procedure or forensics steps** (`operations.md` "Audit log verification
    fails"). The M04 spec explicitly defers key rotation to M14, and M14 has
    not landed an operator runbook.
  - "child agent crashed mid-run" — **no dedicated runbook**. The existing
    crash section in `operations.md` (line 29) is for the *orchestrator*
    crashing, not a child agent dying mid-story. `troubleshooting.md`'s
    `never_active` covers launch failure only.
- **Error code catalog:** documented as a curated subset in
  `docs/cli-reference.md` (lines 188-211, ~12 codes). Source has 30+ codes
  (`rg '"error":\s*"[a-z_]+"' skills/` returns dozens). No central
  `error-codes.*` file, no "suggested action" column. The current table
  declares "additive" but stops short of completeness.
- **State-inspection commands:** 2 of 3 present.
  - `audit-verify` — present.
  - top-level `status` — absent (covered by
    `orchestrator-helper state-summary <file>`, which is awkward but works).
  - `telemetry tail` — absent. `telemetry-report` only aggregates; there is
    no recent-events view.

Finding count: 5 — 4 MUST (F-D10-001 high, F-D10-002 high, F-D10-003 medium,
F-D10-005 low likelihood), 1 SHOULD (F-D10-004).

## F-D10-001
- severity: must
- likelihood: high
- title: No `story-automator doctor` command — operator has no preflight
- file: skills/bmad-story-automator/src/story_automator/cli.py:104-138 (registry absent)
- excerpt: |
    return {
        "version": _cmd_version,
        "derive-project-slug": cmd_derive_project_slug,
        ...
        "audit-verify": cmd_audit_verify,
        "record-cost": cmd_record_cost,
    }
- repro: |
    cd /home/ubuntu/projects/personal/bmad-automator
    PYTHONPATH=skills/bmad-story-automator/src python3 -m story_automator doctor
    # -> "Unknown command: doctor", exit 1, no fallback suggestion.
- proposed_fix: |
    Add `doctor` to the registry. Run ~10 health checks and emit one JSON
    object plus a human-readable summary under `--human`. Minimum check
    list, all observable from existing code paths in `core/`:
    1. `tmux -V` present and ≥ 3.0 (project already shells `tmux capture-pane`
       and `tmux send-keys` in `core/tmux_runtime.py`).
    2. `claude --version` reachable (orchestrator spawns
       `claude --dangerously-skip-permissions`).
    3. `codex --version` reachable (only if `agent-config-presets.json`
       references codex).
    4. `filelock` import + tmpfile lock/release round-trip (used in
       `core/telemetry_emitter.py`).
    5. `psutil` import + `Process(os.getpid())` round-trip.
    6. `resource.getrlimit(RLIMIT_NOFILE)` ≥ 1024.
    7. Free disk on project root > 1 GiB (`shutil.disk_usage`).
    8. Write-permission probe on `_bmad-output/story-automator/` and
       `telemetry/`.
    9. `BMAD_AUDIT_KEY` either unset (warn) or 32+ chars (no echo).
    10. `BMAD_ALLOW_CEILING_BYPASS` unset (must be unset in prod —
        `SECURITY.md:83`).
    11. `agent-config-presets.json` parses (the M04 spec calls this file
        trust-critical).
    Exit nonzero on any hard fail. Emit `{"ok": true|false, "checks":
    [...]}` JSON by default; `--human` adds a green/red one-line summary.
- effort: M
- confidence: 5

## F-D10-002
- severity: must
- likelihood: high
- title: Runbook missing for "child agent crashed mid-run"
- file: docs/operations.md (absent) — N/A — no section covers this scenario
- excerpt: |
    # operations.md has:
    ## Audit log verification fails
    ## A crashed orchestrator is blocking the agent   <-- orchestrator, not child
    ## A corrupt marker
    ## A stuck or wedged child session                <-- hang, not crash
    ## Resuming after an interruption
    ## Inspecting a run's telemetry
- repro: |
    Imagine the operator wakes to a state doc that says story 1.4 is
    IN_PROGRESS, `tmux list-sessions` shows no session for that story
    (it exited nonzero hours ago). What now?
    - `operations.md` "stuck or wedged" assumes the session still exists.
    - `troubleshooting.md` `never_active` assumes the session never started.
    - "Resume after interruption" hand-waves "see State And Resume" with no
      decision tree for "the child crashed AND state is mid-story".
    Operator has no documented path: kill marker? wipe in-progress flag?
    re-spawn? leave alone and let orchestrator detect on next poll?
- proposed_fix: |
    Add `## A child agent crashed mid-run` section to `docs/operations.md`.
    Cover (a) detection signals (no tmux session for active story,
    last-heartbeat timestamp old, `monitor-session` returns `crashed`), and
    (b) a 3-branch decision tree: orchestrator still alive → let it
    auto-retry via the M07/M08 path; orchestrator dead → recover with
    `orchestrator-helper marker check` + `state-update` to roll the story
    back to its last completed step; story unrecoverable → mark deferred
    via `state-update --set status=DEFERRED`. Reference the `commit-ready`
    and `verify-step` helpers as forensics tools.
- effort: S
- confidence: 4

## F-D10-003
- severity: must
- likelihood: medium
- title: Audit-chain runbook lacks key-rotation procedure + forensics steps
- file: docs/operations.md:11-27
- excerpt: |
    ## Audit log verification fails
    ...
    - `valid: false` — the chain broke at/after `last_valid_seq`. ...
      treat the post-`last_valid_seq` history as compromised and re-derive
      state from the git history and the orchestration-state document
      rather than the audit log.
    - `audit_key_missing` — no `BMAD_AUDIT_KEY` in the environment;
      export the run's key and re-run.
- repro: |
    Operator sees `{"valid": false, "last_valid_seq": 17}`.
    Doc tells them post-17 records are suspect and to re-derive state.
    It does NOT tell them:
    - which audit file to archive for forensics, and where (`_bmad/audit/audit.jsonl`?)
    - how to rotate `BMAD_AUDIT_KEY` (start a fresh chain? truncate? archive?)
    - whether to bring the orchestrator down first
    - how to confirm whether the cause was tampering vs. truncation vs. key swap
- proposed_fix: |
    Extend the section with: (a) archive command:
    `cp _bmad/audit/audit.jsonl _bmad/audit/audit-$(date -u +%FT%TZ).jsonl`;
    (b) "is it tampering or truncation?" check — `wc -l` the file,
    inspect tail; (c) key-rotation steps: stop orchestrator, set
    new `BMAD_AUDIT_KEY` env, move old log out of the way, restart;
    (d) reference SECURITY.md's `BMAD_AUDIT_KEY` paragraph. The M04 spec
    deferred this to M14, but a minimal procedure should land before the
    rewrite to unblock the operator today.
- effort: S
- confidence: 4

## F-D10-004
- severity: should
- likelihood: medium
- title: No `telemetry tail` / recent-events view for live debugging
- file: skills/bmad-story-automator/src/story_automator/commands/telemetry_report.py:55-113
- excerpt: |
    def cmd_telemetry_report(args: list[str]) -> int:
        """Entry point for ``story-automator telemetry-report``.
        Optional flags:
            --events <path>   telemetry JSONL ...
            --report {cost_by_epic,attempts_by_story,retro_inputs,all}
- repro: |
    Operator wants to watch the last 20 events to see what the
    orchestrator just did. Today they must `tail -F telemetry/events.jsonl`
    and read raw JSONL because `telemetry-report` only emits aggregations.
    `audit-verify` confirms integrity but doesn't show event sequence.
    `orchestrator-helper state-summary` shows state doc, not event ledger.
- proposed_fix: |
    Add `telemetry-report --tail N` (default 20) that streams the last N
    parsed events as a JSON array, plus `--follow` for `tail -f`-style
    live view. Reuse `TelemetryReader.iter_events`. Strictly read-only,
    no lock, so a live run is unaffected. Alternatively expose as a
    distinct `telemetry-tail` command — either is acceptable for SHOULD.
- effort: S
- confidence: 4
