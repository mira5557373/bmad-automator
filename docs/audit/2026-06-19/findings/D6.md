# D6 — Prompt-injection tripwires (emit-only, trust-contract-compatible)

## Summary

**All 8 named tripwires are missing.** The codebase contains zero prompt-injection-detection infrastructure: no signature file, no scanner module, no telemetry event types, no hypothesis tests. The 13 registered `EVENT_TYPE` values in `skills/bmad-story-automator/src/story_automator/core/telemetry_events.py:132-289` are all lifecycle (`story_started`, `story_completed`, `retry_attempt`, etc.) — none correspond to an observed-anomaly tripwire. There is no `data/injection-signatures.txt` (or analogue), no scanner function reachable from the story-read path (`commands/orchestrator.py:_story_file_status` reads `.md` files and never scans them), and no tests under `tests/` matching `tripwire`, `StoryInjection`, `ToolCallInStory`, `injection`, `jailbreak`, or any of the 8 named events.

This dimension is essentially a green-field gap. Findings are filed per-tripwire so each can be checked off independently; F-D6-001 covers the umbrella infrastructure (signatures file + event-base class + provenance plumbing) that all eight share.

## F-D6-010
- severity: should
- likelihood: medium
- title: No StoryFileMutatedMidAttempt sha-pinning to bound the tripwire-evaluation window
- file: N/A — not implemented (related to but distinct from the 8 named tripwires)
- excerpt: |
    N/A
- repro: |
    # Without `story_sha_at_start` + re-hash on attempt-completion, an attacker
    # (or a buggy child) can rewrite the story file between read and emit,

- proposed_fix: At attempt start, hash the story file and record `story_sha_at_start` on every TripwireEvent and on `StoryStarted`. - effort: M
