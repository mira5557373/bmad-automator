## F-D3-001
- severity: must
- likelihood: high
- title: BMAD_AUDIT_KEY leaks to every child via /proc/<pid>/environ
- file: skills/bmad-story-automator/src/story_automator/core/common.py:108-122 (and twin at core/utils.py:95-122)
- excerpt: |
    def run_cmd(*args: str, timeout: int = DEFAULT_COMMAND_TIMEOUT, env: dict[str, str] | None = None, cwd: str | Path | None = None) -> tuple[str, int]:
        proc_env = os.environ.copy()
        if env:
            proc_env.update(env)
        completed = subprocess.run(
            list(args),
            ...
            env=proc_env,
            ...
        )
- repro: |
    # Parent process exports BMAD_AUDIT_KEY (per audit_verify_cmd, this is the
    # HMAC signing key for the audit chain). run_cmd does os.environ.copy(),
    # so the tmux server, every `tmux send-keys` invocation, every `git`,
    # `pgrep`, `ps` child receives BMAD_AUDIT_KEY in its environ. Same UID =>
    # /proc/<tmux-pid>/environ is readable by the spawned child agent in the
    # pane. The deliberate `-e BMAD_AUDIT_KEY=` scrub at tmux_runtime.py:634
    # only scrubs the *pane*'s env, not the *tmux server process* env that
    # launched it; the child agent can still read the parent tmux's /proc
    # entry while their tmux server is alive.
- proposed_fix: |
    Centralize env scrubbing in run_cmd. Add a module constant
    `_SECRET_ENV_VARS = ("BMAD_AUDIT_KEY",)` and replace
    `proc_env = os.environ.copy()` with a helper that drops those keys
    unless the caller opts in via an explicit allowlist parameter. The audit
    *parent* (audit_verify_cmd) loads the key from os.environ directly and
    doesn't go through run_cmd, so nothing legitimate breaks.

    Diff sketch (common.py and utils.py twin):
    ```python
    _CHILD_ENV_DENYLIST = frozenset({"BMAD_AUDIT_KEY"})

    def _child_env(extra: dict[str, str] | None) -> dict[str, str]:
        base = {k: v for k, v in os.environ.items() if k not in _CHILD_ENV_DENYLIST}
        if extra:
            base.update(extra)
        return base

    def run_cmd(...):
        proc_env = _child_env(env)
        completed = subprocess.run(list(args), env=proc_env, ...)
    ```
- effort: S
- confidence: 5

## F-D3-002
- severity: should
- likelihood: medium
- title: LLM parser prompt embedded in argv, leaking via /proc/<pid>/cmdline
- file: skills/bmad-story-automator/src/story_automator/commands/orchestrator_parse.py:44-52
- excerpt: |
    prompt = _build_parse_prompt(contract, parse_contract, "\n".join(lines))
    result = run_cmd(
        str(parser_cfg["provider"]),
        "-p",
        "--model",
        str(parser_cfg["model"]),
        prompt,
        env={"STORY_AUTOMATOR_CHILD": "true", "CLAUDECODE": ""},
        timeout=int(parser_cfg["timeoutSeconds"]),
    )
- repro: |
    # Any process on the box (same UID) can read /proc/<pid>/cmdline while
    # the parser child runs. The prompt contains up to 150 lines of the
    # child agent's prior output — operationally non-secret but the project
    # already pipes prompts via stdin in spec_compliance.py for exactly this
    # reason ("clean environment via -p with stdin" pattern is documented).
- proposed_fix: |
    Mirror spec_compliance.py: pass the prompt via stdin instead of argv.
    Requires teaching run_cmd to accept an `input=` parameter (subprocess.run
    already supports it) or using subprocess.run directly here:
    ```python
    completed = subprocess.run(
        [str(parser_cfg["provider"]), "-p", "--model", str(parser_cfg["model"])],
        input=prompt,
        capture_output=True, text=True,
        timeout=int(parser_cfg["timeoutSeconds"]),
        env=_child_env({"STORY_AUTOMATOR_CHILD": "true", "CLAUDECODE": ""}),
        check=False,
    )
    ```
- effort: S
- confidence: 4

## F-D3-003
- severity: should
- likelihood: low
- title: Two near-duplicate run_cmd helpers risk drift on security hardening
- file: skills/bmad-story-automator/src/story_automator/core/common.py:108 and skills/bmad-story-automator/src/story_automator/core/utils.py:95
- excerpt: |
    # common.py:108
    def run_cmd(*args: str, timeout: int = DEFAULT_COMMAND_TIMEOUT, env: dict[str, str] | None = None, cwd: str | Path | None = None) -> tuple[str, int]:
        proc_env = os.environ.copy()
        ...
    # utils.py:95
    def run_cmd(*args: str, cwd: ..., env: ..., timeout: ...) -> CommandResult:
        merged_env = os.environ.copy()
        ...
- repro: |
    The two helpers differ only in return type (tuple vs CommandResult) and
    in a small FileNotFoundError branch in utils.py. Any hardening (env
    scrubbing per F-D3-001, stdin support per F-D3-002, executable
    canonicalization) must be applied in both or one of the codepaths will
    silently regress.
- proposed_fix: |
    Pick one helper as canonical (utils.py is the richer one — has the
    TimeoutExpired and FileNotFoundError branches the tmux probes rely on)
    and have common.run_cmd be a thin tuple-returning adapter that calls
    utils.run_cmd. Or vice-versa. One enforcement point for env hygiene.
- effort: S
- confidence: 4

## F-D3-004
- severity: nice
- likelihood: low
- title: claude/codex/tmux/git rely on PATH lookup — doctor should pin paths
- file: skills/bmad-story-automator/src/story_automator/core/spec_compliance.py:251,283 and run_cmd callers passing bare "tmux"/"git"/"claude"
- excerpt: |
    def check_compliance(*, ..., claude_binary: str = "claude", ...):
        ...
        completed = subprocess.run([claude_binary, "-p"], ...)
    # plus run_cmd("tmux", ...), run_cmd("git", ...), run_cmd("pgrep", ...)
    # at dozens of sites
- repro: |
    Single-user threat model rules out a malicious co-tenant, but a stale
    shim in ~/.local/bin or /usr/local/bin (e.g. an `ai-tool` installer
    that drops a `claude` wrapper) silently takes over. The basic.py
    doctor at line 234 already checks `shutil.which("tmux")`; extending
    that to record the resolved path and warn on subsequent drift is a
    cheap belt-and-suspenders win.
- proposed_fix: |
    In `basic.doctor`, log the absolute resolved paths for claude, codex,
    tmux, git, bash on success — and refuse to start orchestration if any
    binary changes path between invocations (cache in the state dir). Not
    blocking; document in operator docs that PATH integrity matters.
- effort: M
- confidence: 3

## F-D3-005
- severity: nice
- likelihood: low
- title: run_cmd inherits LANG/LC_* uncontrolled — parsing of child output may vary
- file: skills/bmad-story-automator/src/story_automator/core/common.py:109 and utils.py:101
- excerpt: |
    proc_env = os.environ.copy()
    if env:
        proc_env.update(env)
- repro: |
    spec_compliance.py:279 explicitly sets `"LANG": "C.UTF-8"` on the
    claude child to guarantee deterministic locale, but run_cmd callers
    (tmux probes, git commands) get whatever locale the parent has. A
    user with `LC_ALL=tr_TR.UTF-8` triggers Turkish-locale `lower()` /
    sort ordering inside `git`, `tmux` output that parsing code may not
    expect. Not a security finding; correctness/portability finding worth
    flagging here since it's the same env-hygiene codepath.
- proposed_fix: |
    Set `LC_ALL=C.UTF-8` (or at least `LANG=C.UTF-8`) inside run_cmd by
    default unless the caller overrides. One-line addition in the same
    helper consolidation per F-D3-003.
- effort: S
- confidence: 3
