## K-2 outcome (cache wired, hot readers, hit rate observability)

Commit `32032cb` — `perf(evidence): K-2 — bundle memoization with explicit invalidation on persist`.

**What it adds**

- New sibling module
  `skills/bmad-story-automator/src/story_automator/core/evidence_cache.py`
  (~120 LOC, well under the 500-LOC soft cap; isolated from
  `evidence_io.py` so the cache lifecycle is independently testable).
  Public surface:
  - `cached_load_evidence_bundle(project_root, gate_id) -> list[dict]`
    — cache-aware wrapper around the real loader.
  - `invalidate_evidence_cache(project_root, gate_id)` — single-key
    drop; the only invalidation called by production code.
  - `invalidate_all_evidence_cache()` — bulk drop intended for test
    isolation; does NOT bump the per-key invalidation counter so the
    counter remains a faithful signal of in-band invalidations.
  - `evidence_cache_stats() -> dict[str, int]` — snapshot of `hits` /
    `misses` / `invalidations` for observability and tests.
- `evidence_io.persist_evidence_record` now calls
  `invalidate_evidence_cache(project_root, gate_id)` after every
  successful write. That is the **single source of cache invalidation**
  — no TTL, no size cap, no background refresh. A persist is the only
  in-process event that can mutate a bundle, so this is the complete
  fresh-after-write contract.
- Two hot readers swapped from `load_evidence_bundle` to
  `cached_load_evidence_bundle`:
  - `verdict_engine.evaluate_gate` — the verdict entry point.
  - `gate_orchestrator` — both the Merkle-root export site and the
    `fail_closed` error collector.
  Both readers see the same `gate_id` within one `run_production_gate`
  call, so the second read is now a cache hit.

**Why it matters**

- Eliminates the redundant disk scan + per-record schema validation on
  the second and third invocation inside a single gate run — the
  exact cost shape called out by the round-3 deferred list.
- Adds observability (`evidence_cache_stats()`) so future regressions
  (e.g. a fresh code path that bypasses the wrapper) can be detected by
  a test asserting hit-count grew.
- Avoids touching the frozen gate surface — the wrapper is a sibling
  module and the original `load_evidence_bundle` is unchanged, so any
  audit-replay tool that imports it directly still works.

**Correctness boundary**

- Returned bundles are `copy.deepcopy`'d so callers that mutate the
  result cannot poison subsequent cache hits.
- Cache misses load **outside** the lock — a slow disk read does not
  block concurrent readers of unrelated `gate_id`s. Two concurrent
  misses on the same key may both call `load_evidence_bundle`; the
  result is deterministic and idempotent, so the worst case is a
  duplicated read, never a corrupted cache.
- Out of scope by design: persists from **other processes**. Each
  process owns its own cache; cross-process freshness is the
  filesystem's job (the fsync-parent durability bar landed in M-3 still
  applies).

**Test coverage**

- New `tests/test_evidence_cache.py` (~176 LOC, 17 tests) covers:
  - Hit / miss / invalidation counters increment correctly.
  - Persist invalidates exactly the affected `(project_root, gate_id)`
    pair.
  - Mutating a returned bundle does not affect the cached copy
    (deep-copy contract).
  - `Path` and `str` `project_root` inputs collide to the same key.
  - `invalidate_all` is bulk-only and does not bump per-key counter.
  - Concurrent miss on the same key is safe (last-writer-wins
    determinism).

## C2 MVP outcome (8-genre lineage ledger, what is in scope vs deferred)

Commit `cdb61c7` — `feat(innovation): C2 MVP — cross-genre artifact lineage ledger (brainstorm → gate)`.

**What it adds**

- New module
  `skills/bmad-story-automator/src/story_automator/core/innovation/lineage_ledger.py`
  (~280 LOC, well under the 500-LOC soft cap; intentionally a *sibling*
  of M54 `ledger.py`, not an extension — M54 chains NFR evidence
  inside one gate, C2 chains artifacts across genres).
- Closed-vocabulary genre tuple in canonical pipeline order:
  ```
  LINEAGE_GENRES = (
      "brainstorm", "braindump", "brief", "BRD",
      "PRD", "kernel", "story", "gate",
  )
  ```
  `verify_lineage(expected_genres=LINEAGE_GENRES)` can assert a full
  brainstorm → gate chain when needed.
- `LineageEntry` dataclass — one record-of-record per pipeline
  artifact, with `parent_root` referencing the Merkle root of all
  preceding entries.
- `LineageChain` — the validated sequence; constructed via
  `build_lineage_chain(entries)` which walks the input list, verifies
  each `parent_root` equals the running root over the preceding
  entries, and raises `LineageError` on any mismatch.
- `verify_lineage(chain)` — boolean predicate that detects payload
  tampering by recomputing every entry hash + running root.
- `find_orphans(entries, present_roots)` — surfaces entries whose
  claimed parent is not in the supplied set; the primitive that future
  disk-persistence + operator CLI will use for "show me dangling
  lineage" queries.
- `LineageError` (a `ValueError` subclass) is the single exception
  type for unknown-genre / broken-chain / tampered-payload cases.

**Hash discipline (mirrors M54)**

- Canonical JSON: `sort_keys=True`, compact separators
  (`","` / `":"`), no whitespace.
- SHA-256 over the resulting UTF-8 bytes.
- Determinism is the entire safety story — entries in input order
  define the root; reordering changes it. Tests cover the
  order-sensitivity invariant explicitly.

**In scope for the MVP (shipped today)**

- Pure in-memory chain construction + verification.
- 8-genre closed vocabulary.
- Hash discipline byte-compatible with the M54 ledger style so a future
  cross-ledger audit tool can reuse the same canonical-JSON helper.
- 17 focused unit tests in `tests/test_lineage_ledger.py` covering:
  build / verify / tamper-detection / unknown-genre rejection /
  order-sensitivity / orphan detection / empty-chain edge / single-entry
  / multi-genre happy path.

**Out of scope (deferred to follow-up milestones, documented at
module docstring tail)**

- Disk persistence under `_bmad/lineage/<chain_id>/`.
- Gate-file embedding — the gate file will eventually carry a
  `lineage_root` field so a verdict carries proof of its provenance.
- Operator CLI to query the lineage of a specific gate.
- Operator-facing visualization (graphviz / mermaid output).
- Cross-process / cross-machine chain merging.

Each deferred item is a distinct follow-up milestone (C2.1 … C2.5) so
the MVP can land cleanly without inviting scope creep. The M54 ledger
went through the same staged adoption and the discipline paid off.

**Why it matters**

- First concrete deliverable in the C-class innovation lane —
  everything that came before in the C lane was instrumentation or
  scaffolding.
- Establishes the artifact-provenance primitive the rest of the
  innovation lane builds on: brainstorm-to-gate traceability is the
  prerequisite for "why did this PRD ship?" replay tooling.
- Sibling-not-extension lets M54 stay frozen at its current surface
  — the two ledgers solve different problems and entangling their
  schemas would have made both harder to evolve.
